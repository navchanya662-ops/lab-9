# Лабораторна робота 9

## Тема

Реєстрація користувачів: `bcrypt`, валідація email, збереження користувача в MongoDB.

## Мета роботи

Мета лабораторної роботи - реалізувати безпечну реєстрацію користувачів у Node.js/Express API. У проєкті використано MongoDB для збереження даних, Mongoose для опису моделей, `bcryptjs` для хешування паролів та MVC-структуру для організації коду.

Ця робота є продовженням попереднього API для блог-платформи. До вже наявної структури було додано модуль авторизації та модель користувача.

## Варіант проєкту

Обраний варіант: блог-платформа.

Основна сутність варіанту: `Article`.

Поля сутності `Article`:

- `title` - заголовок статті;
- `content` - текст статті;
- `author` - автор;
- `tags` - теги;
- `publishedAt` - дата публікації.

## Для чого зроблено проєкт

Проєкт демонструє базовий серверний функціонал, потрібний для реєстрації користувачів:

- приймання JSON-запиту від клієнта;
- перевірка обов'язкових полів;
- перевірка формату email;
- перевірка збігу `password` і `confirmPassword`;
- перевірка унікальності email;
- хешування пароля перед збереженням;
- збереження користувача в MongoDB;
- повернення відповіді без поля `password`.

Пароль не зберігається у відкритому вигляді. Перед записом у базу він перетворюється на bcrypt-хеш.

## Основні файли ЛР9

### `app.js`

Головний файл запуску сервера. У ньому:

- підключається Express;
- підключається MongoDB;
- підключаються маршрути;
- налаштовується обробка JSON;
- підключається глобальний обробник помилок.

Маршрут реєстрації підключено так:

```js
app.use('/api/auth', authRoutes);
```

### `models/User.js`

Mongoose-модель користувача.

Поля:

- `name` - ім'я користувача;
- `email` - унікальний email з перевіркою формату;
- `password` - хешований пароль, не повертається у звичайних запитах через `select: false`;
- `role` - роль користувача, `user` або `admin`;
- `createdAt` - дата створення.

### `controllers/authController.js`

Контролер реєстрації користувача.

Він виконує:

- перевірку наявності всіх полів;
- перевірку довжини пароля;
- перевірку збігу пароля та підтвердження;
- пошук користувача з таким email;
- хешування пароля через `bcrypt.hash`;
- створення нового користувача;
- повернення відповіді без пароля.

### `routes/authRoutes.js`

Файл маршруту авторизації.

```js
router.post('/register', register);
```

Повний endpoint:

```text
POST /api/auth/register
```

### `models/Article.js`

Модель основної сутності для варіанту блог-платформи. Для цієї лабораторної потрібно створити лише схему з полями та валідацією, без окремих роутів.

## API реєстрації

### Успішна реєстрація

Запит:

```http
POST http://localhost:3000/api/auth/register
Content-Type: application/json
```

Тіло запиту:

```json
{
  "name": "Ivan Petrenko",
  "email": "ivan@example.com",
  "password": "password123",
  "confirmPassword": "password123"
}
```

Очікувана відповідь:

```json
{
  "success": true,
  "message": "Реєстрація успішна",
  "data": {
    "user": {
      "id": "...",
      "name": "Ivan Petrenko",
      "email": "ivan@example.com",
      "role": "user",
      "createdAt": "..."
    }
  }
}
```

Поле `password` у відповіді відсутнє.

### Повторний email

Якщо користувач з таким email уже існує, сервер повертає помилку `409`.

```json
{
  "success": false,
  "message": "Користувач з таким email вже існує",
  "data": null
}
```

### Відсутнє поле

Якщо не передано одне з обов'язкових полів, сервер повертає помилку `400`.

```json
{
  "success": false,
  "message": "Усі поля обовʼязкові",
  "data": null
}
```

### Паролі не збігаються

Якщо `password` і `confirmPassword` різні, сервер повертає помилку `400`.

```json
{
  "success": false,
  "message": "Паролі не збігаються",
  "data": null
}
```

Для тестування підготовлено файл:

```text
test.http
```

У ньому є запити для перевірки:

- успішної реєстрації;

- відсутнього поля;
- різних паролів;
- некоректного email.

Також ці самі запити можна виконати в Postman.

## Що перевірити в MongoDB

Після успішної реєстрації у колекції користувачів має з'явитися документ.

Потрібно перевірити:

- email збережено;
- роль має значення `user`;
- пароль збережено не відкритим текстом, а bcrypt-хешем;
- поля `confirmPassword` у базі немає.

## Формат відповідей

## Висновок

У лабораторній роботі реалізовано реєстрацію користувачів для Express/MongoDB API. Дані користувача проходять базову валідацію, email перевіряється на унікальність, пароль хешується за допомогою `bcryptjs`, а відповідь клієнту не містить чутливих даних. Проєкт організовано за MVC-структурою, що полегшує подальше розширення функціоналу.
